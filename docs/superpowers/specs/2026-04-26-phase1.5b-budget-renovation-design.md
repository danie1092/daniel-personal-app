## 배경

홈에서 KPI만 보고 자세한 지출 내역은 가계부 페이지에서 본다는 사용자 의도. Phase 1에서 홈은 토스 스타일로 정리됐지만 가계부는 아직 옛 디자인. 매일 보는 곳이고 Phase 3에서 SMS 자동입력이 들어올 예정이라 디자인 + Server Components 마이그레이션을 먼저 끝낸다.

기존 기능은 모두 보존:
- 3탭(입력/세부내역/월별요약)
- 13개 카테고리 + 4개 결제수단 + 고정지출 일괄 추가
- 도넛 차트 + 카테고리 분포
- SMS 클립보드 파싱(`/api/budget/auto`) — Phase 3에서 자동화될 endpoint를 입력 탭에서 수동으로도 활용 (paste → parse → 폼 자동 채움)

## 목표

1. 페이지를 토스 스타일로 재디자인 (홈 KPI 카드와 동일한 시각 언어)

## 결정 사항 (자동 진행 — 사용자 위임)

### 디자인 결정
- **3탭 구조 그대로** (입력 / 세부내역 / 월별요약)
- **기본 탭 = 세부내역** — 사용자가 페이지 진입 시 가장 보고 싶은 것은 "이번 달 얼마 썼나"
- **상단 헤더**: "가계부" 제목 + 월 선택 pill (`‹ 2026년 4월 ›`)
- **세부내역 탭**: 월 KPI 카드 → 카테고리 chip 필터 → 일자별 그룹 리스트 → "빠른 입력" 하단 버튼
- **입력 탭**: 큰 금액 input → 카테고리 그리드(13개, 4×4 마지막 줄 1개 제외) → 결제수단 chip → 설명/메모/날짜 → SMS 붙여넣기 영역 → 저장 버튼
- **월별요약 탭**: 도넛 차트 + 카테고리별 막대 + 월급/저축 카드 + 고정지출 일괄 추가 버튼
- **수정/삭제 흐름**: 행 탭 → 시트 슬라이드업 → 수정/삭제

// 모든 Action은 첫 줄에 await requireSession() 후 실패시 ok=false 반환

  category: BudgetCategory;
  description: string;   // ≤ 200

  amount: number;        // 정수

export async function updateBudgetEntry(id: string, input: ...): Promise<...>;
export async function deleteBudgetEntry(id: string): Promise<...>;
export async function addFixedExpenses(yearMonth: string): Promise<{ ok: true; added: number; skipped: number } | ...>;
export async function parseAndCreateFromSms(rawText: string): Promise<{ ok: true; entry: ... } | ...>;  // /api/budget/auto 로직 흡수 (선택)

`parseAndCreateFromSms`는 옵션 — Phase 3에서 외부 호출 endpoint(`/api/budget/auto`)는 그대로 두고, 내부 paste 사용은 Server Action으로 일원화 가능. 단순함을 위해 **Phase 1.5b에서는 기존 `/api/budget/auto` fetch 호출 패턴 유지** (입력 탭에서 paste → fetch endpoint → 폼 자동 채움). Server Action으로의 흡수는 Phase 3에서.

### URL state 정책

- 월: `?ym=2026-04` (없으면 현재 월)
- 탭: `?tab=details|input|summary` (기본 details)
- 카테고리 필터(세부내역): `?cat=식사` (없으면 전체)

URL state로 관리하면 새로고침/공유/뒤로가기 모두 자연스럽고, Server Component가 URL을 읽어 데이터 fetch.

## 보안 / 검증 패턴 (Phase 0/1 follow-up 반영)

- 모든 Server Action은 `requireSession()` 호출
- 입력 검증: amount(0 ≤ x ≤ 999_999_999, 정수만), description(≤ 200), memo(≤ 500), paymentMethod(PAYMENT_METHODS 화이트리스트 또는 null), category(BUDGET_CATEGORIES 화이트리스트), date(`YYYY-MM-DD` 정규식 + 합리적 범위 1990~2100)
- `console.error("..."`, err instanceof Error ? err.message : "unknown")` — raw error 노출 X
- service role key 사용 X — 모두 SSR client (`@/lib/supabase/server`)
- RLS가 사용자 격리 — 추가 user_id 필터 불필요 (단, 단일 사용자라 영향 없음)

## 위험 / 트레이드오프

- **URL state 동기화** — 탭/월/필터를 URL에 두면 좋지만 클라이언트 상태와 동기화가 까다로움. Next.js 16의 `searchParams` prop을 적극 활용 + `router.replace()` 사용.
- **Server Action revalidatePath 비용** — 매 mutation 후 `/budget` revalidate. 단일 사용자라 부하 무시.
- **도넛 차트 server 렌더** — SVG는 server에서 렌더 가능. 인터랙티브(hover) 필요하면 client 컴포넌트로 분리. Phase 1.5b에선 정적 SVG로 충분.
- **`/api/budget/auto` endpoint의 무인증 + module-level supabase init** — Phase 0에서 식별된 위험이지만 Phase 3 영역. 지금 손대지 않음. (Phase 3 design doc에 인계)
