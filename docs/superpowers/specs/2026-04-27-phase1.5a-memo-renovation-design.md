## 배경

메모는 사용자가 빠르게 한 줄씩 떠오르는 것을 적는 곳. 홈에서 빠른메모로 한 줄 입력 가능(Phase 1)하지만, 메모 페이지는 검색/필터/태그/그리드 뷰가 있는 풀 인터페이스. Phase 1에서 [메모|큐레이션] 탭 분기만 추가했고 메모 탭 내용은 옛 디자인 그대로. 이제 토스 스타일로 갈아엎고 Server Components 전환.

채집함(인스타 자동수집 후 미정리 항목 + AI 정리 시트)은 Phase 2에서 자동화될 예정이지만 현재 흐름(채집함 버튼 → 시트 → AI 정리 → 저장)은 그대로 유지.

큐레이션 탭은 Phase 1의 placeholder 그대로 둠 — Phase 2에서 본격 구현.

## 결정 사항 (자동)

### 디자인
- **헤더**: "메모" 제목 + 채집함 버튼(미처리 카운트 chip)
- **탭 헤더**: [메모 | 큐레이션] 그대로 유지 (Phase 1 패턴)
- **입력 카드**: 큰 textarea + 태그 chip picker + 저장 버튼. 토스풍 카드 안에 배치
- **검색 + 필터**: 검색 input + 태그 chip 가로 스크롤 (전체/태그별)
- **메모 그리드**: 2열 그리드, 태그별 파스텔 배경(`memoColors.ts` 그대로 사용), 콘텐츠 line-clamp 5, 태그 + timeAgo
- **채집함 시트**: 슬라이드업, 미정리 URL 리스트 → "AI로 정리" 버튼 → 정리된 그룹 → "저장" 버튼. 기존 흐름 보존, 디자인만 토큰화

### Server Actions
```ts
createMemo({ content, tag }): { ok, id } | error
updateMemo(id, { content, tag }): { ok } | error
deleteMemo(id): { ok } | error
organizeInbox(): { ok, groups } | error                  // 기존 /api/inbox/organize 로직 흡수
saveInboxGroups(groups): { ok } | error                  // 기존 /api/inbox/save 로직 흡수
```

`/api/inbox/organize`, `/api/inbox/save` API endpoint는 그대로 둠 — 외부에서 호출하지 않으니 Phase 2에서 정리 가능. 단 Server Action으로 흡수하면 클라이언트 fetch 호출 제거 + 보안 일관성 ↑. **결정: Server Action으로 흡수**.

### URL state
- `?tab=memo|curation` (기본 memo) — Phase 1 패턴 유지

검색/필터는 URL state로 두지 않음 (메모 탭은 자주 진입 + 검색은 일시적, URL 권한이 너무 큼).

### 입력 검증
- content: 1~8192자, trim
- tag: MEMO_TAGS 화이트리스트
- 그룹 입력(saveInboxGroups): MAX_GROUPS=50, 각 group의 content ≤ 8192, tag 화이트리스트, item_ids 배열

## 위험 / 트레이드오프
- **organizeInbox Server Action에서 Anthropic 호출** — 기존 API endpoint 그대로 두고 fetch로 호출하는 방법도 있지만, Server Action으로 흡수하는 게 일관성 ↑. Anthropic 토큰은 server-side만 접근.
